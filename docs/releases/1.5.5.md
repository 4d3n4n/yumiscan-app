# v1.5.5 — Mise a jour des modeles IA et hardening Assistant IA

Date: 2026-03-17

Status: implementation locale

## Constat

Au 17 mars 2026, le pipeline scan et l'assistant utilisaient encore `gemini-2.0-flash` en dur dans les Edge Functions.

Probleme:

- Google retourne maintenant une erreur `404` sur ce modele pour les nouveaux projets / nouveaux usages
- le back-office ne permettait pas de changer rapidement de modele sans redeployer
- les KPI finance etaient encore libelles autour de `Gemini 2.0 Flash`

Constat externe pris en compte:

- la page officielle de deprecation Gemini recommande `gemini-2.5-flash` comme remplacement de `gemini-2.0-flash`
- la famille `Gemini 3 Flash` reste presente en preview, donc non retenue comme defaut prod pour cette passe

References externes:

- https://ai.google.dev/gemini-api/docs/deprecations
- https://ai.google.dev/gemini-api/docs/changelog
- https://ai.google.dev/models/gemini
- https://ai.google.dev/pricing

## Objectifs

- supprimer le hardcode `gemini-2.0-flash`
- rendre le modele du scan configurable depuis le back-office
- rendre le modele de l'assistant configurable depuis le back-office
- faire evoluer les couts theoriques admin vers des cles explicites par usage
- conserver une marge admin coherente avec le perimetre de cout reellement mesure

## Decision produit et technique

Defaut retenu:

- `scan_ai_model = gemini-2.5-flash`
- `assistant_ai_model = gemini-2.5-flash`

Pourquoi:

- `gemini-2.5-flash` est le remplaçant officiel stable le plus naturel pour ce repo
- le pipeline scan reste sensible a la qualite des sorties JSON et de nettoyage OCR
- la preview `Gemini 3 Flash` n'est pas le meilleur choix par defaut pour un flux prod critique

## Changements implementes

### 0. Assistant IA guide par la classification du scan

L'Assistant IA ne part plus seulement de listes plates comme `blocking_labels_clean`.

Il construit maintenant une `evidence matrix` directement depuis le `result_json` du scan:

- `direct_blockers`
- `ambiguous_scoped`
- `ambiguous_global`

Chaque evidence conserve:

- l'ingredient detecte
- sa `reason`
- le ou les criteres utilisateur concernes
- sa `scope`

Impact produit:

- un ingredient non-ok ne peut plus etre attribue a un autre critere utilisateur
- une ambiguite globale est formulee comme un doute transversal, pas comme un blocage certain
- les fallbacks et prompts utilisent la meme source de verite metier

Exemples corriges:

- `soja + halal`, preuve `soja` -> l'assistant ne doit parler que de `soja`
- `halal`, preuve `porc` -> l'assistant parle de compatibilite `Halal`
- ingredient flou global -> l'assistant parle de doute sur l'ensemble des criteres selectionnes

Impact technique:

- la signature cache Assistant IA a evolue
- les anciennes fiches assistant sont considerees stale via la nouvelle version de cache
- des tests dedies couvrent maintenant les cas `allergene`, `profil`, `ambigu scoped`, `ambigu global`

### 0.1. Contrat de cache Assistant IA aligne front / backend

Le scan etant immuable une fois persiste, le sujet n'etait pas un `scan` qui change, mais un contrat de cache Assistant IA qui n'etait pas valide de la meme maniere cote front et cote backend.

Correctifs ajoutes:

- `ASSISTANT_CACHE_VERSION` est maintenant partage entre modules front et backend
- le front refuse une fiche cachee si:
  - la `version` est obsolete
  - `source_signature` est absente
  - `generated_at` est absente
  - la `response` ne respecte pas le contrat attendu
- le merge local du cache ecrit toujours la version courante
- `append_cards` cote backend ne repart plus d'une simple "derniere reponse", mais d'une fiche valide pour la signature source courante

Impact UX:

- on evite d'hydrater une fiche assistant stale apres evolution de pipeline
- on evite d'enrichir une fiche invalide avec de nouvelles cartes
- la confiance percue de l'utilisateur augmente, sans ajouter de latence reseau

### 0.2. UX mixte pour les scans avec blocage + ambiguite secondaire

L'Assistant IA expose maintenant un etat UX `mixed` cote front quand un scan contient a la fois:

- des ingredients bloquants
- et des ingredients encore ambigus a verifier

Important:

- le contrat backend ne change pas
- `mode` reste `allergen` ou `ambiguous`
- `mixed` est seulement une couche de presentation front

Impact UX:

- le CTA inline ne parait plus trop binaire
- la sheet peut dire clairement `X bloquants + Y a verifier`
- l'utilisateur comprend mieux que le produit est deja bloque, tout en voyant qu'il reste aussi des zones floues a faire clarifier

But produit:

- reduire le stress en voyage
- eviter qu'un scan mixte paraisse "juste ambigu" ou "juste bloque"
- donner une lecture plus calme et plus honnete de la situation

### 0.3. Refactor code propre de `scan-ambiguous-assistant`

La fonction edge `scan-ambiguous-assistant` a ete decoupee en sous-modules:

- `context.ts`
- `cache.ts`
- `helpers.ts`
- `fallbacks.ts`
- `gemini.ts`
- `schema.ts`
- `types.ts`

But:

- rendre le handler HTTP lisible
- isoler les responsabilites metier
- limiter la complexite cognitive du fichier principal
- faciliter les tests et les evolutions futures

Important sur les "phrases en dur":

- elles existent toujours cote backend
- elles ne sont pas du copy front
- elles servent de fallbacks deterministes et de garde-fous quand Gemini echoue, renvoie un JSON invalide ou produit une sortie jugee incoherente

Depuis le hardening suivant:

- le prompt Gemini contient maintenant un exemple explicite du JSON attendu
- si Gemini renvoie un JSON invalide, le backend lance une seconde passe de "repair JSON" aupres de Gemini
- le fallback deterministe n'est utilise qu'en dernier recours, si la reparation echoue aussi

Decision retenue:

- on conserve ces fallbacks, car ils securisent l'UX en voyage
- on les centralise dans `fallbacks.ts` pour expliciter leur role et eviter de les melanger au handler HTTP

### 0.4. Etat final du hardening Assistant IA

Le lot `1.5.5` sur l'Assistant IA couvre maintenant quatre niveaux:

- normalisation metier a partir du `result_json`
- prompt Gemini contraint avec exemple explicite du JSON attendu
- passe de reparation JSON par Gemini si la premiere sortie est invalide
- fallback deterministe backend en dernier recours seulement

Le code final de l'edge function est maintenant organise autour de modules distincts:

- `index.ts` pour le handler HTTP
- `context.ts` pour le chargement du contexte scan / config
- `cache.ts` pour la signature source et la persistence du cache
- `gemini.ts` pour les prompts, l'appel modele et la reparation JSON
- `fallbacks.ts` pour les reponses de secours deterministes
- `helpers.ts` pour la normalisation, la comparaison et la sanitization
- `schema.ts` / `types.ts` pour le contrat local

Etat attendu en production:

- la plupart des fiches doivent etre generees par Gemini
- les fallbacks peuvent encore apparaitre, mais beaucoup plus rarement qu'avant
- si un fallback apparait, il doit rester coherent avec le scan et plus rassurant qu'une erreur vide

### 1. Runtime scan

Le scan ne lit plus le modele depuis un hardcode.

Il resolve maintenant le modele dans cet ordre:

1. `app_config.scan_ai_model`
2. `SCAN_AI_MODEL`
3. `GEMINI_MODEL`
4. fallback interne `gemini-2.5-flash`

Impact:

- un changement de modele scan peut etre pilote depuis `/app/admin/settings`
- les phases `0`, `0.5`, `0.9`, `1.5` et `2` utilisent toutes ce meme modele runtime

### 2. Runtime assistant

L'assistant IA ne lit plus le modele depuis un hardcode.

Il resolve maintenant le modele dans cet ordre:

1. `app_config.assistant_ai_model`
2. `ASSISTANT_AI_MODEL`
3. `GEMINI_MODEL`
4. fallback interne `gemini-2.5-flash`

Impact:

- l'assistant peut evoluer separement du pipeline scan si besoin
- on peut tester un modele assistant plus economique plus tard sans toucher au scan principal

### 3. Back-office admin

Le back-office expose maintenant:

- `scan_ai_model`
- `scan_batch_size`
- `assistant_ai_model`
- `google_ocr_cost_eur_per_request`
- `scan_ai_cost_eur_per_request`
- `assistant_ai_cost_eur_per_request`

Le scan peut donc maintenant etre pilote depuis le BO sur deux axes distincts:

- le modele IA
- la taille des lots envoyes a `Phase 1.5` / `Phase 2`

Regle retenue:

- `scan_batch_size` est borne cote backend entre `1` et `16`
- la valeur par defaut reste `8`
- l'objectif est de tester le meilleur compromis entre latence, cout, stabilite et risque d'hallucination

Le champ historique `gemini_flash_cost_eur_per_request` reste seulement en legacy/fallback technique pour ne pas perdre la valeur deja saisie avant migration.

### 4. KPI finance

La marge observee continue de s'appuyer sur le pipeline scan, car ce sont les seules requetes dont les compteurs sont persistés par scan:

- `ocr_request_count`
- `gemini_request_count`

La formule observee devient donc:

- cout observe = `OCR Google` + `IA scan`
- marge observee = `revenus - cout observe`

Important:

- le cout `assistant_ai_cost_eur_per_request` est configurable dans le BO
- mais il n'entre pas encore dans la marge observee faute de compteur d'usage assistant persiste dans `scans`

Ce choix evite de melanger:

- un cout reellement mesure
- et un cout assistant encore non instrumente

### 5. Observabilite performance scan

Chaque scan finalise persiste maintenant un `performance_json` leger dans `scans`.

But:

- suivre la latence reelle du pipeline sans activer `scan_debug_enabled`
- remonter ces donnees dans le back-office admin
- comparer rapidement le scan le plus court, moyen et le plus long sur une periode

Le dashboard admin expose desormais:

- temps le plus court
- temps moyen
- temps le plus long
- moyennes de quelques blocs utiles (`Phase 0`, `Phase 0.5`, `Classification`, `Finalisation`)

Important:

- ces stats couvrent les scans persistés avec metriques completes
- elles ne dependent pas du `debug_json`
