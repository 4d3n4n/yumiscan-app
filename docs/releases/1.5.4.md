# v1.5.4 — Assistant IA Japanese TTS

Date: 2026-03-16

Status: implementation locale

## Scope

Cette passe vise a ajouter la lecture audio des phrases japonaises dans la fenetre `Analyse IA`.

Objectifs produit:

- permettre a l'utilisateur de faire ecouter directement une phrase japonaise generee par l'assistant
- garder une UX fluide dans la sheet Assistant IA
- eviter un surcout inutile si l'utilisateur ne lance jamais l'audio
- ne pas degrader la latence de generation de l'assistant existant

## Contexte actuel

Le repo dispose deja des briques suivantes:

- l'assistant IA est genere cote backend par [scan-ambiguous-assistant/index.ts](/Users/adenankhachnane/Downloads/projet-full-stack/main/supabase/functions/scan-ambiguous-assistant/index.ts)
- le backend envoie au modele uniquement le contexte utile:
  - ingredients ambigus
  - ingredients detectes comme bloquants / allergenes
  - allergenes / restrictions actifs de l'utilisateur
- les phrases japonaises sont deja presentes dans `store_phrases[].japanese`
- la reponse assistant est cachee dans `scans.assistant_cache_json`
- le front consomme cette reponse dans [ScanAiAssistantSheet.vue](/Users/adenankhachnane/Downloads/projet-full-stack/main/components/scan/ScanAiAssistantSheet.vue)

Conclusion:

- le texte source TTS existe deja
- il manque la couche audio, son cache et le bouton play

## Contraintes metier a respecter

L'Assistant IA n'est pas un chatbot libre.

Il sert a:

- clarifier un scan ambigu ou bloque
- reformuler les points de vigilance
- aider l'utilisateur a demander une verification ou une alternative

Contrainte produit:

- on n'envoie pas "tout le scan" au modele
- on lui envoie surtout:
  - les ingredients ambigus
  - les ingredients bloques / allergenes
  - les allergenes ou restrictions de l'utilisateur

Donc:

- l'assistant doit raisonner a partir de `signaux classes`
- pas a partir d'un texte libre de conversation

Autre contrainte UX:

- si une fiche assistant existe deja en base pour la langue courante, elle doit s'ouvrir directement
- l'ecran d'introduction avec `Lancer l'analyse IA` ne doit apparaitre que si aucune fiche n'existe encore pour cette langue

Autre contrainte regeneration:

- `Regenerer` doit ecraser la fiche precedente dans `assistant_cache_json`
- et invalider / remplacer les audios TTS des phrases devenues obsoletes

Nouvelle contrainte d'enrichissement:

- l'utilisateur peut demander des cartes supplementaires
- ces cartes s'ajoutent uniquement aux `store_phrases`
- elles sont cumulees en cache jusqu'a un maximum de `6`
- au-dela de `6`, aucun nouvel ajout ne doit etre genere

## Options analysees

### Option 1 — generer les audios des la reponse assistant

Pipeline:
- l'utilisateur ouvre l'assistant
- l'Edge Function genere la reponse IA
- dans la meme passe, elle genere aussi tous les audios japonais
- elle persiste le tout en cache

Avantages:

- lecture quasi instantanee au clic
- un seul flux backend a raisonner
- pas de seconde requete audio au premier play

Inconvenients:

- augmente la latence initiale de l'assistant
- fait payer du TTS meme si l'utilisateur ne clique jamais sur play
- devient vite lourd si on stocke l'audio brut dans `assistant_cache_json`
- complexifie la regeneration de l'assistant, car texte et audio deviennent couples

Verdict:

- bon pour une UX premium
- moins bon pour le cout et la performance initiale

### Option 2 — generer l'audio au clic, sans cache audio

Pipeline:
- l'utilisateur clique sur play
- une Edge Function relit le texte japonais depuis le cache assistant
- elle appelle Google Cloud TTS
- elle renvoie l'audio au front

Avantages:

- pas de surcout tant que l'utilisateur ne joue pas l'audio
- latence initiale de l'assistant intacte
- implementation simple

Inconvenients:

- premier play plus lent
- l'audio est regenere a chaque clic si rien n'est persiste
- pas ideal si l'utilisateur relance plusieurs fois la meme phrase

Verdict:

- meilleure que l'option 1 pour un premier shipping rapide
- encore imparfaite si on veut un usage repete propre

### Option 3 — recommandee: generation a la demande + cache audio dedie

Pipeline:
- l'utilisateur clique sur play
- une Edge Function dediee lit la phrase japonaise depuis `assistant_cache_json`
- elle cherche un cache audio par `scan + phrase + hash du texte + voice`
- si le cache existe, elle renvoie l'URL signee
- sinon elle genere l'audio, le stocke, puis renvoie l'URL signee

Avantages:

- pas de cout TTS inutile tant qu'il n'y a pas de clic
- pas de ralentissement de la generation initiale de l'assistant
- les replays sont rapides
- separation claire entre cache texte et cache audio

Inconvenients:

- une Edge Function supplementaire
- un peu plus de plomberie (storage + metadata)

Verdict:

- meilleur compromis pour ce repo
- c'est l'option recommandee pour `1.5.4`

## Recommandation retenue

Recommandation:

- ne pas stocker l'audio binaire dans `assistant_cache_json`
- ajouter une generation TTS a la demande
- ajouter un cache audio dedie, persistant et invalidable
- utiliser Google Cloud Text-to-Speech avec une voix japonaise **WaveNet**

Pourquoi:

- `assistant_cache_json` doit rester un cache texte/structure
- l'audio peut grossir vite et polluer la ligne `scans`
- le TTS ne doit pas ralentir l'assistant quand l'utilisateur veut juste lire

## Design technique propose

### Backend

Nouvelle Edge Function proposee:

- `scan-assistant-tts`

Input minimal:

- `scan_id`
- `phrase_index`
- `voice_id` optionnel si on veut laisser evoluer les voix plus tard

Regles backend:

- verifier l'utilisateur avec `getAuthUser()`
- verifier que le scan lui appartient
- lire `assistant_cache_json`
- extraire `store_phrases[phrase_index].japanese`
- rejeter si le texte est absent ou vide
- appeler Google Cloud TTS en **WaveNet** pour la generation audio japonaise

### Cache audio

Approche recommandee:

- stocker l'audio dans un bucket Storage dedie, par exemple `assistant-audio`
- garder un manifest leger en base, pas le binaire

Deux options de manifest:

- soit une nouvelle colonne `assistant_tts_cache_json` sur `scans`
- soit une table dediee `scan_assistant_audio_cache`

Preference:

- si on veut ship vite: colonne `assistant_tts_cache_json`
- si on veut quelque chose de plus evolutif: table dediee

Pour `1.5.4`, la colonne JSON peut suffire si on garde une structure simple:

- `language`
- `phrase_index`
- `voice`
- `text_hash`
- `storage_path`
- `generated_at`

### Nettoyage au delete scan

Contrainte obligatoire:

- quand un scan est supprime, tous les audios TTS associes doivent etre supprimes aussi

Regle de design:

- le chemin Storage doit rester clairement rattache au `scan_id`
- exemple recommande:
  - `assistant-audio/{user_id}/{scan_id}/{locale}/phrase-{index}/{voice}/{hash}.mp3`

Pourquoi:

- eviter les fichiers orphelins
- eviter le cout Storage inutile
- garder un nettoyage symetrique avec les images de scan

Implementation attendue:

- lors du delete scan, supprimer tous les fichiers sous le prefixe Storage lie au scan
- exemple: `assistant-audio/{user_id}/{scan_id}/...`
- cette purge par prefixe reste la protection principale si un upload audio a reussi mais que la metadata DB n'a pas pu etre persistee
- nettoyer ensuite la metadata TTS restante

### Cle de cache

Cle stable recommandee:

- `scanId / locale / phraseIndex / voice / textHash.mp3`

Le `textHash` doit dependre de:

- la phrase japonaise
- la voix choisie
- une version interne de pipeline si on change les reglages audio plus tard

### Voix TTS

Choix retenu pour la passe:

- **Google Cloud WaveNet**
- langue: `ja-JP`
- une seule voix japonaise par defaut pour le premier shipping

Pourquoi:

- rendu plus naturel qu'une voix standard
- meilleure experience quand l'utilisateur fait ecouter la phrase a un tiers
- spec plus simple si on ne multiplie pas encore les variantes de voix

Point d'implementation:

- la voix exacte retenue pour cette passe est `ja-JP-Wavenet-B`
- elle entre dans la cle de cache pour invalider proprement les anciens audios si on change plus tard

### Format audio

Choix recommande pour un premier shipping:

- `MP3`

Pourquoi:

- lecture simple cote navigateur
- poids correct
- integration facile avec `HTMLAudioElement`

### Front

Dans [ScanAiAssistantSheet.vue](/Users/adenankhachnane/Downloads/projet-full-stack/main/components/scan/ScanAiAssistantSheet.vue):

- ajouter un bouton play sur chaque bloc japonais
- etat `loading` par phrase
- une seule lecture a la fois
- stopper l'audio courant si l'utilisateur lance une autre phrase
- fallback visible si le TTS echoue
- si une fiche existe deja en cache pour la langue courante, l'afficher directement a l'ouverture
- si aucune fiche n'existe encore, afficher l'introduction avec `Lancer l'analyse IA`
- autoriser l'ajout de cartes supplementaires jusqu'a `6` max sans regenirer tout le resume
- utiliser des actions compactes a icones pour `copier` et `ecouter`
- placer `Regenerer` dans le bloc de synthese de la fiche
- placer `Ajouter des cartes` apres la derniere carte, en CTA pleine largeur

UX recommandee:

- bouton discret avec icone `play / stop`
- spinner pendant le premier fetch
- pas d'autoplay

## Secrets et securite

Le secret Google Cloud TTS doit:

- rester uniquement cote Edge Function
- ne jamais etre expose au client

Point de vigilance:

- si l'API actuelle repose sur une simple API key, c'est acceptable pour un premier backend-only shipping
- mais a moyen terme, un compte de service Google Cloud est plus propre qu'une cle API brute
- le choix **WaveNet** doit rester configure cote serveur, jamais laisse au client sans validation

Secrets supportes dans le code backend:

- `GOOGLE_CLOUD_TEXT_TO_SPEECH_API_KEY`
- `GOOGLE_CLOUD_TTS_API_KEY`
- `GOOGLE_TTS_API_KEY`

Le premier secret non vide trouve est utilise par `scan-assistant-tts`.

## Guardrails

- ne pas ralentir la generation initiale de l'assistant
- ne pas stocker de gros blobs audio dans `assistant_cache_json`
- ne pas exposer l'API key cote client
- ne pas permettre l'acces audio a un scan qui n'appartient pas a l'utilisateur
- ne pas lancer plusieurs audios concurrents cote front
- ne pas laisser d'audios orphelins apres suppression d'un scan

## Decision actuelle

Decision recommandee pour le dev:

- partir sur l'option 3
- utiliser une voix japonaise **WaveNet** comme standard de la passe
- shipping en deux etapes:
  1. Edge Function TTS a la demande + cache audio
  2. bouton play simple dans la sheet

Cela donne la meilleure base pour rester rapide, propre et evolutif.
