# Milestone 1.4 — Performance mobile, cache sélectif et navigation applicative

## Statut
- Version livrée: `1.4.0`
- Statut: `implémentée et stabilisée`
- Objectif atteint: réduire fortement l'effet "site qui recharge" au retour dans l'app mobile/PWA, sans modifier le coeur métier scan/crédits/Stripe/auth

---

## 1. But métier de la milestone
La 1.4 vise à rapprocher YumiScan d'une vraie app mobile:

- retour plus rapide dans l'app
- moins de flash / reset visuel entre pages
- conservation sélective de données déjà connues
- invalidation fine après les événements métier
- transitions de navigation sobres et cohérentes

Le périmètre couvre:
- front public
- app connectée
- PWA / service worker

Le périmètre exclut volontairement:
- logique scan IA/OCR
- logique Stripe métier
- logique d'authentification
- logique crédits métier

## 2. Ce qui a été livré

### 2.1 Cache PWA / runtime caching
Implémenté dans [nuxt.config.ts](/Users/adenankhachnane/Downloads/projet-full-stack/main/nuxt.config.ts).

Livré:
- cache `StaleWhileRevalidate` pour chunks JS/CSS
- cache `StaleWhileRevalidate` pour images publiques
- cache pour manifest/icônes
- cache pour fonts Google
- pas de fallback Workbox de navigation sur `"/"` pour éviter de détourner les routes SSR/app
- pas de cache runtime des documents HTML publics: la fiabilité repose sur les assets + données applicatives, pas sur un faux mode SPA document
- refresh silencieux de la registration SW au retour dans l'app/PWA pour accélérer la prise en compte d'un SW corrigé
- en dev local, reset automatique des anciens service workers pour éviter que des builds PWA précédents ne polluent la navigation

Objectif:
- accélérer le retour dans l'app
- améliorer le premier rendu après une visite récente
- garder un comportement conservateur sur les zones sensibles

### 2.2 Persistance sélective Vue Query via IndexedDB
Implémenté dans:
- [plugins/vue-query.ts](/Users/adenankhachnane/Downloads/projet-full-stack/main/plugins/vue-query.ts)
- [utils/query-cache.ts](/Users/adenankhachnane/Downloads/projet-full-stack/main/utils/query-cache.ts)
- [plugins/zz-query-auth-boundary.client.ts](/Users/adenankhachnane/Downloads/projet-full-stack/main/plugins/zz-query-auth-boundary.client.ts)

Livré:
- persistance durable de certaines queries seulement
- scoping des snapshots utilisateur par `userId`
- TTL courts/moyens selon le type de donnée
- sanitation avant écriture
- restauration filtrée avant `hydrate()`
- réhydratation suivie d'une revalidation discrète pour les queries concernées
- purge du cache persistant au changement de session ou logout
- bootstrap auth optimiste depuis une session locale fraîche pour éviter de rebloquer `/app/*` au retour dans l'app
- les actions de navigation app ne déclenchent plus de validation auth distante bloquante au clic; elles s'appuient sur le user local ou, au pire, sur `supabase.auth.getSession()` local
- cache de session utilisateur pour les signed URLs d'images de scan, afin d'éviter leur resignature visuelle à chaque navigation

Persisté durablement:
- `user-profile`
- `credits`
- `allergens`
- `stripe-order-history`
- `entitlements`
- `user-scans`
- `scan-detail`
- `scan-allergen-names`
- `app-config`
- `company-config`

Explicitement exclus:
- auth/session
- queries admin
- signed URLs
- scans `processing`
- détail scan `processing`
- pricing public durable

Important:
- les signed URLs restent exclues de la persistance durable IndexedDB
- elles sont seulement conservées dans un cache `sessionStorage` borné à l'utilisateur courant et au cycle de session navigateur/PWA
- le but est d'éviter le refetch visuel des miniatures sans transformer des URLs privées en cache longue durée

## 3. Correctifs ajoutés pendant la stabilisation 1.4
La milestone n'a pas été seulement implémentée, elle a aussi été stabilisée après tests réels.

### 3.1 Persistance query durcie
Problèmes rencontrés:
- `DataCloneError` IndexedDB sur objets non clonables
- risque de réhydrater d'anciens snapshots devenus non autorisés
- risque de fuite locale inter-comptes sur navigateur partagé

Correctifs:
- normalisation profonde des `queryKey`
- sanitation des payloads persistés
- écriture IndexedDB tolérante aux erreurs
- version snapshot bumpée à `3`
- purge automatique des anciens snapshots
- revalidation de chaque query restaurée contre la policy actuelle avant `hydrate()`
- isolation des queries user par `userId`
- rejet de toute rehydratation user si la session courante ne correspond pas au snapshot

### 3.2 Pricing public remis en SSR stable
Problème rencontré:
- mismatch d'hydration sur la home

Correctif:
- `pricing-offers-public` sorti de la persistance durable
- chargement SSR via `useAsyncData` dans [components/home/HomePricing.vue](/Users/adenankhachnane/Downloads/projet-full-stack/main/components/home/HomePricing.vue)

Résultat:
- rendu home plus stable
- plus de workaround d'hydration fragile
- client Supabase non mutualisé entre requêtes SSR

### 3.3 Cleanup service worker ciblé
Problème rencontré:
- le nettoyage Workbox pouvait être trop agressif
- un fallback de navigation Workbox sur `"/"` pouvait provoquer `non-precached-url`, latence puis page blanche après reprise PWA
- en local, un ancien SW pouvait encore contrôler `localhost` et fausser complètement les tests de navigation

Correctif:
- plugin [plugins/00-legacy-sw-cleanup.client.ts](/Users/adenankhachnane/Downloads/projet-full-stack/main/plugins/00-legacy-sw-cleanup.client.ts) réécrit pour:
  - préserver le SW courant `/sw.js`
  - supprimer uniquement les registrations legacy
  - forcer `update()` sur le SW sain si nécessaire
  - ne nettoyer les caches que si une registration legacy a vraiment été supprimée
- config PWA durcie dans [nuxt.config.ts](/Users/adenankhachnane/Downloads/projet-full-stack/main/nuxt.config.ts):
  - `navigateFallback: null`
  - suppression du cache runtime `public-documents`
- plugin [plugins/01-pwa-sw-refresh.client.ts](/Users/adenankhachnane/Downloads/projet-full-stack/main/plugins/01-pwa-sw-refresh.client.ts) ajouté pour appeler `registration.update()` au montage et au retour de visibilité
- plugin [plugins/00-dev-sw-reset.client.ts](/Users/adenankhachnane/Downloads/projet-full-stack/main/plugins/00-dev-sw-reset.client.ts) ajouté pour unregister les SW en dev local

### 3.4 Navigation app rendue non bloquante
Problèmes rencontrés:
- clics nav `/app/*` retardés par une revalidation auth réseau
- warning Vue Router sur `#pricing`
- risque d'écran blanc accentué par les transitions `out-in`

Correctifs:
- [components/app/AppNavigation.vue](/Users/adenankhachnane/Downloads/projet-full-stack/main/components/app/AppNavigation.vue) n'attend plus `resolveAuthenticatedSession()` au clic
- lecture locale `supabase.auth.getSession()` seulement si l'état auth n'est pas encore prêt
- anti-double-clic simple avec `navigationPending`
- [app/router.options.ts](/Users/adenankhachnane/Downloads/projet-full-stack/main/app/router.options.ts) attend la présence réelle de la cible avant scroll hash
- transitions Nuxt conservées en mode `default` pour ne pas sérialiser le montage/démontage pendant les guards

### 3.5 Transitions de page réalignées avec le rendu réel
Problèmes rencontrés:
- le compromis précédent en `mode: default` supprimait les écrans blancs mais ne produisait pas de vraie transition perceptible
- les tentatives CSS seules donnaient un effet intermédiaire: peu d'animation, léger flicker, continuité visuelle insuffisante
- le comportement devait rester strictement front, sans fetch supplémentaire ni impact sécurité

Correctifs:
- ajout du plugin [plugins/03-view-transitions.client.ts](/Users/adenankhachnane/Downloads/projet-full-stack/main/plugins/03-view-transitions.client.ts) pour déclencher `document.startViewTransition()` uniquement côté client et uniquement sur navigateurs compatibles
- attente de `page:finish` avant clôture de la transition, avec timeout défensif pour éviter tout blocage si une page tarde à finir son rendu
- typage des transitions selon la route: `page-public`, `page-soft`, `page-app`, `page-reduced`, plus les cas directionnels `push-app` et `pop-public`
- neutralisation des transitions CSS concurrentes dans [assets/css/globals.css](/Users/adenankhachnane/Downloads/projet-full-stack/main/assets/css/globals.css) quand la View Transitions API est active
- fallback CSS réduit à une transition quasi nulle sur les navigateurs sans support, afin d'obtenir un comportement binaire et stable: vraie transition ou pas de transition

Résultat:
- rendu de navigation plus proche d'une app native sur navigateurs compatibles
- disparition du faux effet de transition qui clignotait
- aucun appel backend supplémentaire
- aucun affaiblissement des guards, rôles ou permissions

Résultat:

## 4. Garde-fous sécurité et scalabilité

### Sécurité
La 1.4 garde les contraintes suivantes:
- aucune persistance durable des queries admin
- aucune rehydratation de queries user hors session correspondante
- aucune persistance durable des signed URLs privées
- aucune persistance durable des scans encore en `processing`
- purge des queries persistées lors d'un changement de session
- `/auth/confirm` exclu du cache document PWA
- cleanup SW limité au callback auth problématique

### Scalabilité
Le design reste scalable car:
- la policy de cache est centralisée dans un seul fichier
- l'invalidation métier est centralisée
- la persistance ne dépend pas d'un composant particulier
- les TTL sont explicites et ajustables
- le système reste extensible par query key sans dupliquer la logique

Limite actuelle de scalabilité front:
- certains chunks client restent trop gros
- la 1.4 améliore le ressenti, mais pas encore le découpage JS global
