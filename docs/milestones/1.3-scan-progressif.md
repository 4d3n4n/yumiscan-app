# Milestone 1.3 — Scan progressif, payload front minimal et debug BO conditionnel

## 1. Objectif de la milestone
Cette milestone 1.3 a un double objectif :

1. **Réduire fortement le temps perçu du scan**
   - aujourd'hui, un scan complet peut prendre environ `15 à 20 secondes`
   - l'utilisateur attend trop longtemps un résultat “tout ou rien”
   - même quand le pipeline fonctionne correctement, l'UX paraît lente

2. **Arrêter d'exposer au front les données internes inutiles**
   - le `result_json` actuel contient encore des structures de debug du pipeline
   - ces données ne sont pas nécessaires au parcours utilisateur
   - elles alourdissent la réponse, compliquent le front, et exposent des détails internes qui doivent rester back/admin

Le but est donc de :
- **garder exactement la même logique métier**
- **rendre le scan progressif côté UX**
- **rendre `result_json` strictement front-safe**
- **déporter le debug dans `debug_json`**
- **ne persister le debug que si le BO l'autorise**

---

## 2. Ce que cette milestone ne doit pas changer
Cette milestone **ne doit pas modifier le coeur métier** du produit.

Elle ne doit pas changer :
- les phases fonctionnelles du pipeline
- la logique OCR
- la logique de traduction
- la logique de mapping allergène
- la logique de classification Gemini
- la logique de reconstruction finale
- la logique de verdict final du produit
- les règles métier de crédits :
  - `free`
  - `daily`
  - `paid`

On change uniquement :
- le découpage du traitement après `flattenTree`
- la façon de persister le scan pendant son traitement
- la forme du payload front
- la façon d'afficher la progression côté `/app/scan/:id`
- la gestion du debug

---

## 3. Décisions retenues

### 3.1 Version
- la version du projet doit passer en **`1.3.0`**
- le footer/version affichée doit suivre
- `package.json` et le lockfile doivent être alignés

### 3.2 Batching progressif
- après `flattenTree`, les ingrédients sont découpés en **batches de 8**
- ces batches sont exécutés avec une **concurrence maximale de 2**
- chaque batch passe dans :
  1. traduction (`phase15`)
  2. mapping allergène déterministe (`phase1.7`)
  3. classification IA (`phase2`)

### 3.3 Affichage progressif
- `food-scan-analyze` doit répondre **rapidement** avec un `scan_id`
- le front doit naviguer immédiatement vers **`/app/scan/:id`**
- la page `/app/scan/:id` doit poll la ligne `scans`
- pendant le traitement, le front doit montrer :
  - les ingrédients déjà traités
  - la progression par lots
  - des placeholders / skeletons / loaders pour les lots manquants

### 3.4 Debug
- le debug pipeline ne doit **jamais** être exposé au front utilisateur
- il doit être stocké dans `debug_json`
- il ne doit être persisté **que si** `scan_debug_enabled = true` dans le BO

### 3.5 Atomicité crédits
- le scan doit être créé immédiatement en `processing`
- le crédit ne doit être consommé **qu'à la fin**
- si le scan échoue avant finalisation :
  - aucun crédit ne doit être consommé

### 3.6 Compatibilité scans historiques
- les anciens scans doivent continuer de s'afficher
- les helpers front/admin doivent :
  - préférer le nouveau format top-level
  - fallback vers l'ancien format `debug.phase7.ingredient_tree`

## 4. Problème actuel du `result_json`
Aujourd'hui, le `result_json` contient encore des données internes de pipeline, du type :
- `debug.phase0`
- `debug.phase05`
- `debug.phase1`
- `debug.phase1_flattened`
- `debug.phase1_5`
- `debug.phase1_7`
- `debug.phase2`
- `debug.phase7`
- etc.

Conséquences :
- payload trop lourd
- dépendance front à des structures internes
- exposition inutile de détails backend
- difficulté à faire évoluer le pipeline sans casser le front

Le principe de la v1.3 est :
- **`result_json` = uniquement ce que l'UI user doit afficher**
- **`debug_json` = uniquement pour debug/admin**

---

## 5. Contrat de données cible côté DB

### 5.1 Extensions de la table `scans`
La table `scans` doit être étendue avec :

- `processing_status text not null default 'completed'`
  - valeurs autorisées :
    - `processing`
    - `completed`
    - `failed`
- `processing_error text null`
- `debug_json jsonb null`

`result_json` reste présent, mais change de rôle :
- avant : payload mixte front + debug
- après : payload **strictement front-safe**

### 5.2 Nouveau contrat `result_json`
Le `result_json` doit contenir uniquement :

- `product_status`
- `ok_ingredients`
- `ambiguous_ingredients`
- `allergens_ingredients`
- `ingredient_tree`
  - **présent uniquement quand le scan est finalisé**
- `meta`

### 5.3 Structure cible de `meta`
```json
{
  "detected_language": "fr",
  "phases_completed": ["0", "0.5", "1", "1.5", "1.7", "2"],
  "is_final": false,
  "batch_progress": {
    "completed_batches": 1,
    "total_batches": 4,
    "completed_items": 8,
    "total_items": 27
  }
}
```

### 5.4 Contenu attendu de `debug_json`
Quand `scan_debug_enabled = true`, `debug_json` peut contenir :
- `phase0`
- `phase05`
- `phase09`
- `phase1`
- `phase1_flattened`
- `phase1_5`
- `phase1_7`
- `phase2`
- `phase7`

Quand `scan_debug_enabled = false` :
- `debug_json = null`

### 5.5 Règle absolue
- `debug_json` ne doit jamais être nécessaire au front utilisateur
- l'UI user ne doit jamais parser ni afficher `debug_json`
- l'UI admin peut l'utiliser **seulement si** le toggle BO est activé

## 7. Pipeline backend détaillé

### 7.1 Phases inchangées
Les phases suivantes restent identiques :
- `phase0`
- `phase0.5`
- `phase0.9` (si nécessaire)
- `phase1`

Le point de bascule de la milestone est **après** `flattenTree`.

### 7.2 Sortie attendue de `flattenTree`
Après `flattenTree`, on dispose d'une liste plate d'ingrédients avec ordre stable.

Chaque nœud aplati doit conserver :
- `id`
- `order`
- `raw`
- assez d'information pour rebrancher ensuite l'arborescence finale

Cette liste plate devient la base des batches.

### 7.3 Découpage en batches
Découper la liste plate en batches de `8` :

```ts
const batches = chunkArray(flatNodesWithOrder, 8)
```
