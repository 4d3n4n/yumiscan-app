# 1.5.4 — Analyse qualite Assistant IA et regeneration

Date: 2026-03-16

Statut: analyse produit et technique, avec hardening principal implemente

## Mise a jour implementation

Depuis la redaction initiale de cette analyse, plusieurs points ont ete traites dans le code:

- `Regenerer` remplace bien la fiche precedente dans `assistant_cache_json`
- l'ouverture de la sheet affiche directement la fiche deja presente en base pour la langue courante
- la reponse persiste maintenant aussi des metadata de generation (`generated_at`, `source_signature`)
- le cache TTS est reconcilie quand les phrases changent
- l'utilisateur peut enrichir `store_phrases` avec des cartes supplementaires, dans la limite de `6`

Depuis la `1.5.5`, une seconde evolution structurante a ete ajoutee:

- l'Assistant IA ne reconstruit plus sa logique metier depuis des listes plates
- il derive maintenant une `evidence matrix` directement a partir du `result_json` du scan
- cette matrice distingue:
  - `direct_blockers`
  - `ambiguous_scoped`
  - `ambiguous_global`
- chaque evidence conserve:
  - l'ingredient concerne
  - son `reason`
  - les criteres utilisateur touches
  - la `scope` (`one`, `many`, `all_selected`)

Depuis les derniers correctifs de `1.5.5`, deux points UX / robustesse supplementaires ont aussi ete traites:

- alignement du contrat de cache Assistant IA entre front et backend
- ajout d'un mode UX `mixed` cote front pour les scans avec blocage + ambiguite secondaire
- refactor de la function edge en sous-modules lisibles (`context`, `cache`, `helpers`, `fallbacks`, `gemini`, `schema`, `types`)
- ajout d'un flux `prompt JSON exemple -> validation -> repair JSON Gemini -> fallback final`

Important:

- les phrases "en dur" qui restent dans le backend ne sont pas du texte front par defaut
- elles correspondent aux fallbacks deterministes utilises quand la sortie Gemini est absente, invalide ou rejetee par le post-sanitization
- ce choix est volontaire pour garder une reponse utile et rassurante meme en cas d'echec modele
- l'objectif actuel est justement qu'elles apparaissent le moins possible, grace a la passe de repair JSON avant fallback

Etat final de la passe clean code:

- le handler principal a ete reduit a l'orchestration HTTP
- les responsabilites ont ete sorties dans des sous-modules dedies
- les vraies erreurs TypeScript du refactor ont ete corrigees
- les checks utiles (`tests` cibles + `deno check`) sont verts

Ce qui peut encore apparaitre dans certains IDE releve surtout de suggestions Sonar de maintenabilite, pas d'un risque runtime bloqueur.

Le coeur du travail restant sur cette piste n'est donc plus la coherence cache/TTS.
Il porte surtout sur:

- la qualite semantique des signaux envoyes au modele
- la qualite du fallback
- la qualite percue des phrases generees

## Etat actuel du pipeline

Depuis les derniers correctifs, la pipeline reelle de l'Assistant IA fonctionne ainsi:

1. lecture du cache valide pour la langue courante si `source_signature` correspond
2. sinon appel Gemini avec prompt renforce et contexte structure
3. validation stricte du JSON modele
4. post-sanitization ligne par ligne cote backend
5. remplacement partiel par fallback quand une ligne est jugee faible ou incoherente
6. persistance dans `assistant_cache_json`
7. reconciliation du cache TTS si les phrases japonaises changent

En `1.5.5`, l'etape `2` ne recoit plus seulement des labels utilitaires. Elle recoit une representation normalisee:

- `direct_blockers`: ingredients deja classes non-ok et rattaches a un critere exact
- `ambiguous_scoped`: ingredients ambigus qui ne concernent qu'un sous-ensemble des criteres selectionnes
- `ambiguous_global`: ingredients ambigus non attribuables proprement, donc a traiter comme doute transversal

Important:

- le post-sanitization n'est pas une seconde IA
- c'est une couche de regles deterministes cote backend
- elle decide si une ligne Gemini est acceptable ou si la ligne fallback correspondante doit la remplacer

## Ce qui a ete effectivement corrige

### 1. Comparaison ancienne / nouvelle fiche

La regeneration ne compare plus simplement du JSON brut.

Le backend compare maintenant une signature normalisee basee sur:

- `mode`
- `risk_level`
- `analysis_summary`
- `ambiguity_reasons`
- `checkpoints`
- `store_phrases`
- `disclaimer`

La normalisation supprime le bruit de forme:

- casse
- espaces
- diacritiques
- ponctuation non utile

But:

- eviter de considerer comme "nouvelle" une fiche qui n'a change qu'a la marge
- relancer une tentative Gemini si la regeneration reste trop proche

### 2. Nettoyage des labels avant usage

Les labels bruts du scan sont maintenant nettoyes avant d'etre reutilises dans le texte.

Exemples de transformations:

- `contient du porc` -> `porc`
- `contains pork` -> `pork`
- `ingredient: lait` -> `lait`

But:

- eviter les resumes laids ou trop litteraux
- mieux ancrer les textes sur des mentions lisibles par un humain

### 3. Separation profil alimentaire vs allergene

La pipeline distingue maintenant explicitement:

- `selected_profile_labels`
- `selected_allergen_labels`
- `selected_profiles`
- `selected_allergen_criteria`
- les ingredients bloquants detectes
- les ingredients ambigus
- les evidences normalisees avec leur `scope`

But:

- eviter des phrases absurdes du type `sans Halal`
- eviter de parler "d'allergenes" quand le cas reel est un profil type `Halal`, `Vegan`, `Kosher`
- eviter d'attribuer un ingredient a un autre critere que celui pointe par le scan

Exemples cibles:

- bon: `Le scan a releve la mention « soja », qui correspond a votre critere soja.`
- mauvais: `La mention « soja » rend ce produit non compatible avec votre regime Halal.`
- bon en ambigu global: `Le doute principal vient de la mention « sauce ». Tant que ce point n'est pas clarifie, il peut concerner l'ensemble de vos criteres selectionnes.`

### 4. Fallback plus naturel

Le fallback a ete reecrit pour:

- mieux citer les mentions detectees
- eviter les fragments bruts recraches tels quels
- produire des titres FR reellement francais
- mieux distinguer les cas:
  - allergene classique
  - profil alimentaire
  - scan ambigu
  - ambiguite globale portant sur toute la selection utilisateur
  - ambiguite ciblee sur un sous-ensemble des criteres

Exemple de direction visee:

- mauvais: `poulet et contient du porc ont ete detectes...`
- meilleur: `Le scan a releve des mentions comme « porc ». Comme cela ne semble pas compatible avec votre regime Halal, le plus prudent est d'ecarter ce produit.`
- meilleur en mixte: `Le scan a releve la mention « soja », ce qui suffit deja a bloquer le produit pour votre allergene soja. L'etiquette reste aussi floue sur « sauce ».`

### 5. Prompt renforce

Le prompt Gemini contient maintenant:

- des labels nettoyes utiles
- des hints de formulation
- des exemples explicites `bad` / `good`
- la matrice d'evidences du scan (`direct_blockers`, `ambiguous_scoped`, `ambiguous_global`)
- des contraintes fortes sur:
  - la localisation FR
  - l'usage de labels concrets
  - la distinction profil vs allergene
  - la coherence ingredient -> critere -> scope
  - l'interdiction de phrases du type `without Halal`

### 6. Post-sanitization apres Gemini

Apres generation, le backend verifie notamment que la sortie:

- ressemble a une vraie phrase humaine
- ne contient pas de tokens internes
- ne traite pas un profil comme un allergene
- ne recopie pas des fragments bruts comme `contient du ...` dans le resume
- n'utilise pas des generiques du type `mes allergenes` si les allergenes reels sont connus
- ne laisse pas passer des titres anglais dans une sortie FR

Si une ligne echoue:

- on ne jette pas forcement toute la fiche
- on remplace seulement la ligne faible par sa version fallback correspondante

## Impact UX reel

Cette pipeline ne ralentit presque pas l'experience normale.

Pourquoi:

- le post-sanitization est local et tres leger
- le cout dominant reste l'appel reseau a Gemini
- si la fiche est deja en cache et encore valide, on ne recalcul rien

Le seul cas qui peut etre un peu plus lent est `Regenerer`, car:

- une premiere tentative Gemini est faite
- si elle reste trop proche de la fiche precedente, une seconde tentative Gemini peut etre lancee
- si cela reste insuffisant, un fallback de regeneration distinct est utilise

Conclusion UX:

- ouverture normale avec cache: quasi immediate
- premiere generation: cout proche de l'ancienne pipeline
- regeneration forcee: parfois legerement plus lente, mais uniquement pour chercher une variation plus nette

Precisions `1.5.5`:

- le cache front ne rehydrate plus une fiche Assistant IA si sa version ou ses metadata ne sont plus valides
- l'enrichissement `append_cards` ne repart plus d'une fiche stale
- un scan mixte peut maintenant afficher explicitement un contexte du type `X bloquants + Y a verifier`

## Constat

Deux ressentis utilisateur remontent sur l'Assistant IA:

- le bouton `Regenerer` donne souvent l'impression de ne rien changer
- les phrases renvoyees restent parfois trop generiques, repetitives ou peu coherentes avec le scan

L'exemple observe sur un scan ambigu montre des symptomes clairs:

- resume base sur des termes trop vagues: `autre`, `acidifiant`, `epaississant`
- checkpoint incoherent: `Demander si Halal peuvent etre presents...`
- cartes de phrases tres generiques et peu ancrees dans le contexte reel du produit
- titres de cartes partiellement en anglais dans un contexte FR: `Question ingredient`

## Diagnostic

### 1. `Regenerer` bypass bien le cache, mais ne garantit pas une sortie differente

Etat actuel du code:

- le front appelle `scan-ambiguous-assistant` avec `force: true`
- cote backend, `force: true` contourne bien la lecture du cache
- une nouvelle reponse est alors calculee puis persistée dans `assistant_cache_json`

Donc:

- techniquement, `Regenerer` ne relit pas simplement le cache
- mais il peut tout a fait produire un texte tres proche, voire identique, si la meme logique retombe sur les memes formulations
- l'effet de regeneration percu depend donc autant:
  - de la variabilite du modele
  - que de la qualite du fallback
  - et de la qualite des signaux injectes

### 2. L'exemple fourni correspond quasi exactement au fallback local

Le rendu colle de facon tres proche au fallback code en dur:

- `Question ingredient`
- `Question contamination croisee`
- `Question fiche produit`
- les phrases japonaises associees
- la structure des checkpoints

Conclusion:

- dans ce cas, on n'est probablement pas sur une vraie sortie Gemini differenciante
- on est tres probablement sur le `fallback`
- si le fallback est utilise, `Regenerer` ne peut quasiment rien varier, car ce fallback est deterministic

### 3. Le probleme principal n'est pas seulement la regeneration

Le coeur du probleme est plutot:

- la qualite de selection des ingredients "utiles"
- la confusion entre `allergene`, `restriction`, `profil alimentaire` et `ingredient`
- le fait que le fallback soit trop fort et trop generique
- le manque d'observabilite pour savoir si la reponse provient du modele ou du fallback
- l'absence d'ouverture directe de la fiche deja presente en base pour la langue courante

## Rappel des contraintes metier

L'Assistant IA recoit:

- les ingredients avec leur statut (`ambiguous`, `contains_allergen`, etc.)
- les raisons associees quand elles existent
- les allergenes / restrictions actifs de l'utilisateur

Il ne doit pas:

- reinventer un diagnostic libre
- traiter une restriction comme si c'etait un ingredient
- faire croire qu'une verification est encore necessaire si le produit est deja bloque

UX attendue:

- si une fiche Assistant IA existe deja dans `assistant_cache_json` pour la langue courante, on l'affiche directement a l'ouverture de la sheet
- l'ecran avec `Lancer l'analyse IA` ne sert que pour le premier calcul
- `Regenerer` doit remplacer la fiche precedente, pas juste recalculer sans effet visible
- un enrichissement de phrases peut exister en plus, mais il doit:
  - enrichir uniquement `store_phrases`
  - conserver le resume et les checkpoints
  - rester limite a `6` cartes max

## Pourquoi l'exemple est faible

### 1. Les labels d'ingredients retenus sont trop pauvres

Des termes comme:

- `autre`
- `acidifiant`
- `epaississant`
- `assaisonnement`
- `composition incomplete`

ne sont pas exploitables tels quels par un utilisateur.

Probleme produit:

- ils n'aident pas a comprendre le vrai doute
- ils n'aident pas a parler a un vendeur
- ils donnent une impression d'IA "bricolage"

### 2. `Halal` est traite comme si c'etait un ingredient

L'exemple:

- `Demander si Halal peuvent etre presents directement ou en traces`

montre une confusion de modele de donnees.

`Halal` n'est pas:

- un ingredient
- un allergene
- une trace

C'est un critere de compatibilite / profil alimentaire.

Conclusion:

- la pipeline doit separer clairement:
  - `profil / restriction`
  - `ingredient detecte`
  - `ingredient ambigu`
  - `allergene`

### 3. Les cartes de phrases sont trop statiques

Les phrases actuelles du fallback sont utiles comme filet de securite, mais pas comme experience cible.

Problemes:

- elles varient peu d'un scan a l'autre
- elles ne sont pas assez ancrees dans le doute detecte
- elles ressemblent a un kit generique plutot qu'a une aide contextuelle

### 4. La version FR n'est pas entierement localisee

Exemples:

- `Question ingredient`

Cela nuit a:

- la qualite percue
- la coherence de marque
- la confiance utilisateur

## Ameliorations recommandees

## Priorite 1 — rendre la regeneration observable

Objectif:

- savoir si l'utilisateur voit une vraie sortie modele ou le fallback

Recommandation:

- ajouter dans la reponse backend des metadata non UI par defaut:
  - `generation_source: "gemini" | "fallback"`
  - `generated_at`
  - `source_signature`

Benefices:

- debug beaucoup plus simple
- on sait si `Regenerer` est utile ou non
- on peut mesurer la part reelle de fallback

## Priorite 2 — filtrer les termes generiques avant la generation

Objectif:

- ne plus construire les resumes et checkpoints autour de labels trop vagues

Recommandation:

- ajouter une liste de labels a faible valeur a declasser ou exclure:
  - `autre`
  - `acidifiant`
  - `epaississant`
  - `assaisonnement`
  - `arome`
  - `additif`
  - `composition incomplete`
  - `composition non entierement detaillee`

Comportement attendu:

- si aucun ingredient vraiment utile ne ressort, le resume doit parler de:
  - `certains additifs ou assaisonnements non detailles`
  - et non d'une liste bancale de termes generiques

## Priorite 3 — separer strictement restrictions et ingredients

Objectif:

- eviter des phrases absurdes du type `si Halal peuvent etre presents`

Recommandation:

- distinguer dans le backend:
  - `selected profile labels`
  - `blocking ingredients`
  - `ambiguous ingredients`
  - `detected allergens`

Regle:

- un label de profil ne doit jamais etre injecte dans une phrase comme s'il etait un ingredient

## Priorite 4 — faire des cartes plus contextualisees

Objectif:

- sortir du trio de cartes quasi identique sur tous les scans

Recommandation:

- construire les cartes a partir du type de doute dominant

Exemples de familles de cartes:

- ingredient non detaille
- assaisonnement / sauce non detaille
- additif ou epaississant d'origine incertaine
- verification de traces
- demande d'alternative plus sure

Regle UX:

- 2 a 3 cartes max
- chaque carte doit repondre a un usage distinct
- pas de carte "remplissage"

## Priorite 5 — durcir le prompt pour eviter le langage vide

Objectif:

- augmenter les chances d'une sortie modele utile

Recommandation prompt:

- interdire explicitement les termes generiques seuls dans le resume
- interdire l'utilisation d'un label de profil comme ingredient
- demander des titres entierement localises
- demander des checkpoints actionnables et non repetitifs
- demander une ancre contextuelle: ingredient, categorie ou raison precise

Exemple de contrainte a ajouter:

- `Do not use generic labels like "other", "acidifier", "thickener" as primary blockers unless they are the only available evidence.`
- `Never treat dietary profile labels such as halal, vegan, kosher or vegetarian as ingredients or allergens.`
- `For French output, all titles must be written in French.`

## Priorite 6 — rendre `Regenerer` plus utile quand le modele marche

Objectif:

- obtenir une vraie variation utile, pas seulement un recalcule identique

Constat:

- avec un prompt stable et une temperature basse (`0.45`), on peut obtenir des sorties tres proches

Recommandations:

- conserver une temperature moderee en mode normal
- utiliser une temperature legerement plus haute sur `force regenerate`
- ou injecter une consigne du type:
  - `Provide an alternative phrasing and card structure while keeping the same facts.`

Important:

- la variation doit porter sur la formulation
- pas sur les faits detectes

## Priorite 7 — ameliorer le fallback lui-meme

Objectif:

- faire en sorte que meme le fallback reste bon

Le fallback actuel doit evoluer:

- titres FR 100% en francais
- checkpoints moins repetitifs
- formulations plus propres pour les scans ambigus
- distinction nette entre:
  - verification d'ingredient
  - verification de traces
  - demande de detail recette / emballage

Le fallback ne doit plus:

- reutiliser des labels trop vagues comme s'ils etaient utiles
- injecter les labels de profil dans des phrases ingredient

## Plan de travail recommande

### Etape 1

- ajouter `generation_source`, `generated_at`, `source_signature`
- logger le taux de fallback

### Etape 2

- filtrer / declasser les labels generiques
- corriger la confusion `profil` vs `ingredient`

### Etape 3

- reecrire le fallback FR/EN
- localiser tous les titres

### Etape 4

- renforcer le prompt Gemini
- tester une variation de temperature uniquement sur `force regenerate`

### Etape 5

- ajouter des tests unitaires de qualite

Exemples de tests a ajouter:

- interdit `Halal` comme ingredient ou allergene dans les checkpoints
- interdit `Question ingredient` en sortie FR
- interdit `autre` comme ingredient principal si d'autres signaux plus utiles existent
- impose des checkpoints distincts et non dupliques
- garantit qu'une regeneration ecrase bien le cache texte et le cache audio

## Conclusion

Le bouton `Regenerer` fonctionne maintenant correctement cote cache et la qualite de sortie a ete nettement durcie par:

- de meilleurs signaux injectes
- une meilleure distinction metier entre profil et ingredient
- un fallback plus naturel
- une couche de post-sanitization deterministic

Le point restant le plus utile pour le debug produit serait maintenant l'observabilite explicite de la source de generation, par exemple:

- `generation_source`
- taux de fallback
- suivi de la part des regenerations qui declenchent une 2e tentative modele

Autrement dit:

- le sujet n'est plus un bug majeur de regeneration
- le sujet devient surtout un travail continu de qualite percue et d'observabilite
